================================================================================
GA4 / GTM AUDIT TOOL — COMPLETE AUDIT REFERENCE
================================================================================

The tool has TWO MODES:
  Foreign site (no config):  python run.py --url https://anysite.com
                             -> runs the 6 infrastructure audits only
  Client site (YAML config): python run.py [--config client.yaml]
                             -> also runs the journeys defined in the config
                                and verifies expected GTM/GA4 IDs are live

Default config.yaml targets the Palm View test ground:
  Site:    https://palm-view-sparkling-water-co-948395720075.asia-south1.run.app
  GTM ID:  GTM-K4BLC735
  GA4 ID:  G-JXWM0TX7FS

Run command:
  python run.py                          # all audits + configured journeys
  python run.py <name> [<name> ...]      # specific audits only
  python run.py --list                   # what's available

Severity levels in output:
  [CRITICAL]  Compliance violation or tracking completely broken
  [HIGH]      Significant data quality or configuration issue
  [MEDIUM]    Implementation gap, informational risk
  [INFO]      Diagnostic output — does not count as a failure

================================================================================
INFRASTRUCTURE AUDITS (no selectors required — run these first)
================================================================================

--------------------------------------------------------------------------------
1. analytics_audit
   Command:  python run.py analytics_audit
   Source:   audit.py → run_analytics_audit()
--------------------------------------------------------------------------------

What it does:
  Opens BASE_URL in Chrome, accepts consent, then interrogates the page
  using three JavaScript executions only — no network proxy.

How each check works:

  GA4 presence
    Scans all <script> src/text and performance.getEntriesByType('resource')
    for the pattern G-[A-Z0-9]{6,}. Also polls resource timing for /collect
    URLs for up to 6 seconds to catch async hits.
    PASS = at least one measurement ID found OR a collect URL appeared.

  Single measurement ID
    Counts distinct G-XXXXXXX IDs found across all scripts and resources.
    FAIL = more than one ID detected (likely duplicate GA4 firing).

  Deployment method
    Priority order:
      1. GTM  — any script src containing "gtm.js", OR
                 window.google_tag_manager object present in JS runtime.
      2. gtag.js — script src containing "/gtag/js", OR
                   window.gtag function present (without GTM object).
      3. Hardcoded — gtag() call found in inline <script> text, no loader.
      4. Unknown — none of the above detected.
    Reports the active GTM container IDs from window.google_tag_manager keys
    (the live runtime object, more reliable than pattern-matching HTML).

  No conflicting implementations
    Checks whether BOTH a GTM loader (googletagmanager.com/gtm.js) AND a
    standalone gtag.js loader are present simultaneously.
    FAIL = both present — high risk of every GA4 event firing twice.

  First-party tagging
    Splits collect URLs and script sources into first-party (hostname is NOT
    google-analytics.com or googletagmanager.com) vs third-party.
    PASS = at least one first-party endpoint found.
    FAIL = all traffic goes directly to Google domains (standard setup,
           but no first-party durability / ITP resilience).

  sGTM / GTG signals
    Looks for: GTM loader served from a non-googletagmanager.com domain,
    collect endpoint on a non-google-analytics.com domain, and any resource
    URL containing "/gtg" or "gtg.json" (Google Tag Gateway).
    PASS = server-side tagging signals detected.
    FAIL = no signals (standard client-side only setup).

--------------------------------------------------------------------------------
2. consent_audit
   Command:  python run.py consent_audit
   Source:   consent.py → run_consent_audit()
   Requires: performance_logging driver (CDP network capture enabled)
--------------------------------------------------------------------------------

What it does:
  Opens BASE_URL WITHOUT clicking accept consent first. This is deliberate —
  pre-consent checks must run before any user interaction. Then accepts
  consent to verify the post-consent behaviour.

  Important: uses the network capture driver (performance_logging=True) so
  it can observe GA4 collect requests in addition to dataLayer events.

How each check works:

  CMP banner detected  [MEDIUM]
    Scans the DOM for visible elements matching known CMP vendor selectors
    in priority order:
      OneTrust, Cookiebot, TrustArc, Quantcast, Didomi,
      ClearCookie, Cookie Script, generic cookie/consent dialog patterns.
    Also checks window.__tcfapi (TCF 2.0 API — vendor-neutral signal).
    PASS = a matching visible element or __tcfapi was found.
    FAIL = no consent UI detected (consent management may be absent).

  Consent Mode implemented  [CRITICAL]
    Reads window.dataLayer and looks for a list-form entry matching:
      ['consent', 'default', { analytics_storage: '...', ... }]
    This is the gtag/GTM Consent Mode v2 default push that must appear
    before any GA4 tags fire.
    PASS = entry found.
    FAIL = not found — Consent Mode is not configured in GTM/gtag.

  Default denied: analytics_storage  [HIGH]
  Default denied: ad_storage          [HIGH]
  Default denied: ad_user_data        [HIGH]
  Default denied: ad_personalization  [HIGH]
    Each of the four Consent Mode v2 signals is checked individually
    against the consent default dict from the check above.
    PASS = signal value is exactly 'denied'.
    FAIL = signal is missing from the dict, or set to 'granted' by default.
    SKIP = only triggered if Consent Mode implemented check failed.

  Pre-consent GA4 firing  [CRITICAL]
    After page load but BEFORE clicking accept, polls the CDP performance
    log for GA4 collect requests (Network.requestWillBeSent events where
    the URL contains /collect and has v=2). Polls for up to 5 seconds.

    A collect request before consent is NOT automatically a violation: with
    Consent Mode in a default-denied state, GA4 sends cookieless pings (gcs
    shows analytics_storage denied) by design — this is compliant. So each
    pre-consent request is inspected by its gcs analytics_storage digit
    (G + version + ad_storage + analytics_storage; e.g. G100 = denied,
    G111 = granted).
    PASS = no collect requests, OR every pre-consent request has
           analytics_storage denied (compliant cookieless pings).
    FAIL = any request fired with analytics_storage granted, or with no
           Consent Mode signal at all (gcs absent) — genuine tracking
           before consent. Reports the event names and gcs values.

  Post-consent update  [HIGH]
    Clicks the accept button (config.consent.accept_button selector), then
    verifies that analytics_storage actually transitions to GRANTED — the real
    proof that consent propagated to Consent Mode.

    Two signals are gathered after the click:
      1. dataLayer consent_update event — polled for up to 8 seconds across
         known event names (consent_update, gtm_consent_update,
         clearcookie_save_preferences, CookieInformationConsentGiven,
         cookieyes-consent-update) and the GTM nested {"value":{"event":...}}
         pattern. This is only a SUPPORTING signal.
      2. gcs state of post-consent collect requests — the AUTHORITATIVE signal.

    The mere existence of post-consent collect requests is NOT proof of a
    grant: Consent Mode keeps sending cookieless (denied) pings regardless, so
    "any collect = pass" would mask a CMP that never grants analytics_storage.

    PASS (authoritative) = a post-consent collect request shows
                           analytics_storage granted (gcs third digit = 1).
                           Reports the denied → granted transition.
    PASS (weak)          = no post-consent collect to inspect, but a
                           consent_update event fired in dataLayer.
    FAIL = collect requests fired post-accept but analytics_storage never
           became granted (CMP not propagating the grant), OR no consent event
           and no collect requests at all (accept selector wrong / CMP not
           integrated). A consent_update event does NOT rescue a still-denied
           gcs.

--------------------------------------------------------------------------------
3. network_audit
   Command:  python run.py network_audit
   Source:   network.py → run_network_audit()
   Requires: performance_logging driver (CDP network capture enabled)
--------------------------------------------------------------------------------

What it does:
  Opens BASE_URL, accepts consent, then captures all GA4 collect requests
  that fired during the session via Chrome DevTools Protocol performance
  logging. Parses each request's URL query params and POST body to extract
  event data, session identifiers, and consent state.

  The CDP log structure Selenium exposes is:
    {"message": {"method": "Network.requestWillBeSent", "params": {...}}}
  Only entries with method=Network.requestWillBeSent, URL containing
  /collect, and query param v=2 are included.
  POST body params (where GA4 sends event details) are merged in.

How each check works:

  GA4 collect requests  [CRITICAL]
    Polls the performance log for up to 10 seconds for any matching
    collect request. All remaining checks are skipped if none are found.
    PASS = at least one collect request observed.
    FAIL = no collect requests — GA4 may not be firing on this page.

  Client ID (cid)  [HIGH]
    Extracts the cid parameter from all collect requests.
    The Client ID identifies a unique browser/device across sessions.
    PASS = cid present; reports the actual value.
    FAIL = cid absent from all requests (identity stitching broken).

  Session ID (sid)  [HIGH]
    Extracts the sid parameter. Session ID groups events into a session.
    PASS = sid present; reports the actual value.
    FAIL = sid absent (session attribution will fail in GA4 reports).

  Consent state (gcs)  [HIGH]
    Extracts the gcs parameter which encodes the Consent Mode state at
    the time each request was sent.
    Format: G + [version flag] + [ad_storage] + [analytics_storage], where
    each signal digit is 1=granted or 0=denied. The first digit after G is a
    version/active flag (always 1 when Consent Mode is on), NOT a signal.
    Only two signals are carried in gcs; ad_user_data and ad_personalization
    are encoded separately in the gcd parameter and are not decoded here.
    Examples:
      G100 = ad_storage denied, analytics_storage denied
      G101 = ad_storage denied, analytics_storage granted
      G110 = ad_storage granted, analytics_storage denied
      G111 = ad_storage granted, analytics_storage granted
    PASS = gcs present; reports raw value and decoded interpretation.
    FAIL = gcs absent — Consent Mode signals not reaching collect calls.

  Event inventory  [INFO]
    Collects all distinct event names (en parameter) across every collect
    request. Reports count and full list.
    Always PASS — this is diagnostic output, not a gate.

  Session timeline  [INFO]
    Groups all collect requests by cid/sid combination and sorts each
    group by CDP timestamp to reconstruct the chronological event order
    within the session.
    Format: [cid=X / sid=Y]  event1 → event2 → event3
    Always PASS — diagnostic output.

DATALAYER JOURNEY AUDITS (defined declaratively in the YAML config)
================================================================================

Journeys are NOT hardcoded — each config file defines its own under
`journeys:` as a list of steps (goto / accept_consent / click / type /
select_index / mark) with `expect` blocks naming the dataLayer event,
required fields (dot-notation), and optional regex `patterns`. The engine
lives in journeys.py (run_journey); the schema is documented in
config.example.yaml. Onboarding a new client site = writing YAML, no code.

The tool polls the dataLayer every 300ms until the expected event appears or
the timeout (15s by default) expires.

PERSISTENT EVENT LOG: core.py injects a recorder into every document that
mirrors each dataLayer push into sessionStorage, which survives same-origin
navigations. So events pushed by onclick handlers immediately before a
full-page navigation (add_to_cart, generate_lead on server-rendered sites)
are still caught — the old "events lost on navigation" gap is fixed.

The after_index pattern: before every interaction step, the engine snapshots
the log length. Each `expect` only matches entries AFTER that index, so stale
events from earlier in the journey are never matched. accept_consent
deliberately does not move the snapshot (page-load events fire before the
banner is dismissed); use a `mark` step to isolate consent-triggered events.

The journeys described below are the ones defined in palm_view_config.yaml
(= default config.yaml) for the Palm View test ground.


--------------------------------------------------------------------------------
Palm View journeys (see palm_view_config.yaml for exact steps + selectors)
--------------------------------------------------------------------------------

  page_load       /            init (client_id, user_properties.customer_type)
                               + page_view (page_title, page_path)
                               KNOWN GAP: site omits customer_type -> FAIL

  consent         /            clearcookie_save_preferences with
                               clearcookie.consentString after accepting the
                               banner (this CMP has no generic consent_update)

  shop            /shop        view_item_list on load; select_item on product
                               tile click
                               KNOWN GAP: tiles don't push select_item -> FAIL

  product_detail  PDP          view_item on load; add_to_cart on button click
                               (event fires just before nav to /cart — caught
                               by the persistent log)

  cart            /cart        view_cart, remove_from_cart, begin_checkout
                               (journey seeds the cart itself via /shop)

  checkout        /checkout    add_shipping_info (ecommerce.shipping_tier).
                               Fires from the Pay button's onclick, so the
                               journey fills the whole form and SUBMITS it —
                               places a test order on the demo site.

  purchase        /order/1     purchase with transaction_id, value, shipping,
                               items; transaction_id must match ^PV-\d+$
                               (regex is a `patterns:` entry in the YAML)

  subscribe       /subscribe   plan-card click pushes select_promotion +
                               generate_lead + sign_up together (all three
                               fire pre-navigation; caught by persistent log)

  contact         /contact     form_start, form_submit_error, form_submit
                               KNOWN GAP: site emits none of these -> FAILs

  search          /shop        search (search_term, search_result_count);
                               the search box only exists on /shop

================================================================================
JOURNEY CONFIGURATION (replaces the old per-selector table)
================================================================================

Selectors now live INSIDE each journey's steps in the YAML config — the only
standalone selector is selectors.consent.accept_button (used by the global
accept_consent helper; common CMPs are auto-detected, so it's optional).

Step types:   goto, accept_consent, mark, click, type, select_index
Assertions:   expect {event, require: [dot.paths], patterns: {path: regex},
              severity, name, timeout}, and skip {name, reason}

Full schema with comments: config.example.yaml (and the journeys.py docstring).
Worked example: palm_view_config.yaml.

To find a selector: open the site in Chrome, right-click the element in
DevTools -> Copy -> Copy selector, paste into the journey step.
